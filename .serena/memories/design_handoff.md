# design_handoff_secondboard/ — design source of truth

Not part of `src/`; a separate handoff package the UI must match. Key files:
- `README.md` — visual spec: exact colors/sizes/spacing per component (§4-§8ish), referenced by section number in code comments (e.g. "README §6.1", "README §4.4").
- `SecondBoard_PROJECT_OVERVIEW.md` — the full product/architecture spec, huge. Notable sections: §6.5/§8.3 (pgn module), §9 (DB tables), §10 (analysis job flow, engine modes/settings/caching), §11 (move classification: centipawn-loss → expected-points, Book/Miss/Great/Brilliant heuristics, reason codes), §12 (Accuracy System — win% sigmoid formula `expected_score = 1/(1+exp(-k*eval_cp))`, `accuracy = 100*exp(-avg_expected_points_loss*scale)`, split per white/black/phase), §13 (strengths/weaknesses insights), §21.1 (**golden rule: Stockfish + deterministic rules decide truth — eval, best move, classification, accuracy; the frontend only renders; never let mock JS or an LLM decide it**), §25.4 (stream long analyses via Tauri events, keep UI responsive).
- `LOGIC.md` — condensed pointers into the reference JS prototype (`reference/logic/*.js`) and which parts are reusable-as-is (`view-math.js` — pure SVG/geometry helpers like `capturedInfo`, `evalGraph`, `radar`, `evalBarPct`) vs. mock-only-replace (`chess-mock.js`, most of `data.js` except the `CLS` classification color/glyph spec, which is kept and lives on as `TOKENS.classification`).
- `reference/screens/*.png` — full-screen mockups per app screen (1-onboarding, 2-analysis tab, 3-review tab, 4-dashboard, 5-openings, 6-insights, 7-training). Use these (not just README prose) to check exact layout/spacing when a screen looks off.
- `reference/logic/*.js` and `reference/piece_sprites/*.svg` — the original prototype's math helpers and piece art; sprites are already wired into `src/lib/board/pieces/`.
- `SecondBoard.dc.html` — the original single-file interactive prototype; some code comments in `review.ts` reference exact line numbers in this file (e.g. "renderVals() (SecondBoard.dc.html lines 1221-1262)") as the ground truth being ported.

Rule of thumb: any new "real data wiring" work (classification, accuracy, insights, etc.) should locate and follow the relevant OVERVIEW §-numbered spec rather than inventing a formula — see `mem:mocks_vs_real` for what's still outstanding against this spec.
